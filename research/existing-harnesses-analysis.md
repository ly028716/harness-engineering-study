# 现有 Harness 项目分析

> 基于开源项目的深入分析，提取关键设计模式和实现策略

## 概述

本文档分析三个主要的开源 Harness 项目：
1. **claude-code-harness** - Plan→Work→Review 循环实现
2. **refact** - 端到端工程任务处理
3. **agent-os** - 规范驱动的开发系统

## 项目一：claude-code-harness

**项目地址**：[Chachamaru127/claude-code-harness](https://github.com/Chachamaru127/claude-code-harness)

### 核心理念

通过自主的 Plan→Work→Review 循环实现高质量开发。

### 架构设计

#### 两智能体协作模式

```
┌─────────────────┐         ┌──────────────────┐
│   Cursor PM     │ ◄─────► │  Claude Code     │
│   (规划管理)     │         │  Worker (执行)    │
└─────────────────┘         └──────────────────┘
        │                            │
        ▼                            ▼
   创建/更新计划              实现具体功能
   评估质量(100分)            编写代码
   列出缺陷任务               运行测试
   改进设计文档               迭代优化
```

**角色分工**：
- **Cursor PM**：项目经理角色
  - 创建和更新计划
  - 自我评估（100分制）
  - 识别缺陷并列出任务清单
  - 改进项目设计文档
  - 执行最终评估

- **Claude Code Worker**：执行者角色
  - 实现具体功能
  - 编写和修改代码
  - 运行测试验证
  - 根据反馈迭代

### 工作流程

#### 1. Plan（规划阶段）

```
输入：任务需求
  ↓
创建初始计划
  ↓
自我评估（100分制）
  ↓
识别缺陷和不足
  ↓
生成改进任务列表
  ↓
更新计划和设计文档
  ↓
最终评估确认
  ↓
输出：完善的执行计划
```

**关键特点**：
- 使用量化评分（100分制）
- 明确的缺陷识别机制
- 任务列表驱动改进
- 多轮迭代直到满意

#### 2. Work（工作阶段）

```
接收计划
  ↓
分解为具体任务
  ↓
逐个实现功能
  ↓
编写测试用例
  ↓
运行验证
  ↓
记录进度
```

#### 3. Review（审查阶段）

```
检查代码质量
  ↓
运行所有测试
  ↓
评估完成度
  ↓
识别问题
  ↓
决策：通过 or 返回 Work
```

### 核心机制

#### 自我评估系统

**评估维度**：
- 计划完整性
- 任务分解合理性
- 依赖关系清晰度
- 风险识别充分性
- 可执行性

**评分标准**：
- 90-100分：优秀，可以执行
- 80-89分：良好，需要小幅改进
- 70-79分：及格，需要明确改进
- <70分：不及格，需要重新规划

#### 缺陷任务列表

**格式**：
```markdown
## 缺陷清单

1. [ ] 缺少错误处理机制
2. [ ] 测试覆盖率不足
3. [ ] 性能考虑不充分
4. [ ] 文档需要补充
```

### 技术实现要点

#### 上下文管理

- 使用符号链接的 `thoughts/` 目录
- 保存所有阶段的中间结果
- 便于跨阶段引用和回溯

#### 工作流分离

**四阶段模型**：
1. **Research**：研究和理解
2. **Plan**：规划和设计
3. **Implement**：实现和编码
4. **Validate**：验证和测试

**阶段间清晰的上下文传递**：
- 每个阶段输出明确的产物
- 下一阶段基于上一阶段的输出
- 避免阶段混淆和任务漂移

### 适用场景

**最适合**：
- 需要高质量保证的项目
- 复杂的功能开发
- 需要明确规划的任务
- 多人协作的场景

**不太适合**：
- 快速原型验证
- 探索性编程
- 简单的脚本任务

### 关键启示

1. **量化评估很重要**
   - 100分制提供明确的质量标准
   - 便于决策是否继续

2. **缺陷驱动改进**
   - 明确列出问题
   - 逐个解决
   - 可追踪进度

3. **角色分离清晰**
   - PM 负责规划和质量
   - Worker 负责执行
   - 避免职责混淆

4. **上下文持久化**
   - 保存所有中间结果
   - 便于回溯和调试
   - 支持长期项目

## 项目二：refact

**项目地址**：[smallcloudai/refact](https://github.com/smallcloudai/refact)

### 核心理念

端到端处理工程任务的 AI Agent，集成开发者工具，规划、执行、迭代直到成功。

### 架构设计

#### 集成式架构

```
┌─────────────────────────────────────────┐
│           Refact Agent Core             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │ Planner │  │Executor │  │Validator│ │
│  └─────────┘  └─────────┘  └─────────┘ │
└──────────────┬──────────────────────────┘
               │
    ┌──────────┴──────────┐
    │                     │
┌───▼────┐          ┌────▼─────┐
│  IDE   │          │   LSP    │
│ 集成   │          │  服务器   │
└────────┘          └──────────┘
    │                     │
    └──────────┬──────────┘
               │
        ┌──────▼──────┐
        │ 开发者工具  │
        │ Git/测试等  │
        └─────────────┘
```

### 核心特性

#### 1. 工具集成

**支持的工具**：
- **IDE 集成**：Sublime Text, VS Code 等
- **LSP 服务器**：提供语言智能支持
- **版本控制**：Git 操作
- **测试框架**：自动运行测试
- **构建系统**：编译和打包

**集成方式**：
- 通过 LSP 协议通信
- 标准化的工具接口
- 插件式架构

#### 2. 端到端任务处理

**完整流程**：
```
接收任务
  ↓
理解需求和上下文
  ↓
制定执行计划
  ↓
调用相关工具
  ↓
生成/修改代码
  ↓
运行测试验证
  ↓
分析结果
  ↓
如果失败：调整策略，重新执行
  ↓
如果成功：完成任务
```

#### 3. 迭代执行机制

**自动重试策略**：
- 分析失败原因
- 调整执行策略
- 重新尝试
- 设置最大重试次数
- 必要时请求人工介入

### 技术实现

#### LSP 服务器

**功能**：
- 代码补全
- 语法检查
- 类型推断
- 重构支持
- 符号导航

**优势**：
- 标准化协议
- 跨编辑器支持
- 丰富的语言特性

#### 工具调用框架

**设计原则**：
- 统一的工具接口
- 异步执行支持
- 错误处理机制
- 结果标准化

### 适用场景

**最适合**：
- 需要深度 IDE 集成的场景
- 多语言项目
- 需要丰富工具支持的开发
- 团队协作开发

**特点**：
- 强大的工具集成能力
- 端到端的任务处理
- 自动化程度高
- 适合复杂工程任务

### 关键启示

1. **工具集成是核心**
   - 不是孤立的 AI
   - 深度集成开发工具
   - 提供完整的开发体验

2. **LSP 协议的价值**
   - 标准化通信
   - 跨编辑器兼容
   - 丰富的语言支持

3. **迭代执行很重要**
   - 不是一次性生成
   - 持续验证和改进
   - 直到任务成功

4. **端到端思维**
   - 从需求到验证
   - 完整的任务闭环
   - 减少人工介入

## 项目三：agent-os

**项目地址**：[buildermethods/agent-os](https://github.com/buildermethods/agent-os)

### 核心理念

规范驱动的开发系统（Spec-Driven Development），为 AI Agent 提供更好的规划和执行能力。

### 架构设计

#### 分层架构

```
┌─────────────────────────────────────────┐
│         命令接口层 (CLI)                 │
│  /plan  /implement  /review  /deploy    │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│         核心工作流层                     │
│  ┌──────────┐  ┌──────────┐  ┌────────┐│
│  │产品规划  │  │功能开发  │  │质量保证││
│  │Workflow  │  │Workflow  │  │Workflow││
│  └──────────┘  └──────────┘  └────────┘│
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│         专业子智能体层                   │
│  ┌────────┐ ┌────────┐ ┌──────────┐    │
│  │架构师  │ │开发者  │ │测试工程师│    │
│  └────────┘ └────────┘ └──────────┘    │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│         配置和标准层                     │
│  编码标准 / 技术栈 / 项目规范           │
└─────────────────────────────────────────┘
```

### 核心特性

#### 1. 规范驱动开发

**核心思想**：
- 先定义规范（Spec）
- 基于规范生成代码
- 规范即文档
- 规范可验证

**规范类型**：
- **产品规范**：产品使命、功能路线图
- **技术规范**：技术栈、架构设计
- **功能规范**：具体功能的详细描述
- **API 规范**：接口定义和契约

#### 2. 模块化工作流

**工作流系统**：
```
Workflow = 指令块 + 配置选项 + 命令
```

**内置工作流**：
- **产品规划工作流**
  - 定义产品使命
  - 规划功能路线图
  - 确定技术栈
  
- **功能开发工作流**
  - 创建功能规范
  - 生成实现代码
  - 编写测试用例
  
- **质量保证工作流**
  - 代码审查
  - 测试执行
  - 性能验证

**工作流组合**：
- 可以组合多个工作流
- 支持自定义工作流
- 工作流间可以传递上下文

#### 3. 编码标准管理

**v3 新特性**：
- 自动发现项目编码标准
- 文档化现有规范
- 强制执行标准
- 持续更新标准

**标准类型**：
- 代码风格
- 命名规范
- 架构模式
- 测试要求
- 文档规范

### 工作流程详解

#### 产品规划阶段

```
1. 定义产品使命
   - 核心价值主张
   - 目标用户
   - 关键指标

2. 规划功能路线图
   - 功能优先级
   - 开发时间线
   - 依赖关系

3. 确定技术栈
   - 编程语言
   - 框架选择
   - 基础设施
   - 工具链

输出：产品规范文档
```

#### 功能开发阶段

```
1. 创建功能规范
   - 功能描述
   - 用户故事
   - 验收标准
   - 技术方案

2. 生成实现代码
   - 基于规范生成
   - 遵循编码标准
   - 包含注释文档

3. 编写测试用例
   - 单元测试
   - 集成测试
   - 端到端测试

输出：可工作的功能代码
```

#### 质量保证阶段

```
1. 代码审查
   - 静态分析
   - 标准检查
   - 安全扫描

2. 测试执行
   - 运行所有测试
   - 覆盖率检查
   - 性能测试

3. 部署验证
   - 构建检查
   - 部署测试
   - 回滚准备

输出：质量报告
```

### 技术实现

#### 命令系统

**核心命令**：
```bash
# 规划
agent-os plan <feature>

# 实现
agent-os implement <spec>

# 审查
agent-os review <code>

# 部署
agent-os deploy <version>
```

#### 配置管理

**配置文件结构**：
```yaml
project:
  name: "项目名称"
  tech_stack:
    - language: "TypeScript"
    - framework: "React"
    - database: "PostgreSQL"

standards:
  coding_style: "airbnb"
  test_coverage: 80
  documentation: "required"

workflows:
  - name: "feature-development"
    steps:
      - spec
      - implement
      - test
      - review
```

### 适用场景

**最适合**：
- 规范化要求高的项目
- 团队协作开发
- 长期维护的产品
- 需要严格质量控制

**特点**：
- 强调规范先行
- 模块化工作流
- 标准化管理
- 适合企业级开发

### 关键启示

1. **规范驱动的价值**
   - 规范即文档
   - 可验证的需求
   - 减少歧义
   - 便于维护

2. **模块化工作流**
   - 可组合
   - 可复用
   - 易扩展
   - 标准化

3. **编码标准自动化**
   - 自动发现
   - 自动执行
   - 持续更新
   - 减少人工成本

4. **分层架构清晰**
   - 命令层
   - 工作流层
   - 智能体层
   - 配置层
   - 职责明确

## 共同设计模式

### 1. 循环迭代模式

所有三个项目都采用某种形式的循环：
- claude-code-harness: Plan→Work→Review
- refact: 执行→验证→调整→重试
- agent-os: 规划→实现→验证→改进

**关键要素**：
- 明确的阶段划分
- 清晰的退出条件
- 失败后的调整机制
- 最大迭代次数限制

### 2. 职责分离模式

不同程度的职责分离：
- claude-code-harness: PM vs Worker
- refact: Planner vs Executor vs Validator
- agent-os: 多个专业子智能体

**好处**：
- 避免角色混淆
- 提高可维护性
- 便于独立优化
- 支持并行执行

### 3. 上下文管理模式

都重视上下文的持久化和传递：
- 保存中间结果
- 阶段间传递信息
- 支持回溯和调试
- 便于长期项目

### 4. 质量门禁模式

多层次的质量检查：
- 语法检查
- 测试验证
- 代码审查
- 性能评估
- 安全扫描

### 5. 工具编排模式

协调多个工具的使用：
- 统一的工具接口
- 标准化的调用方式
- 错误处理机制
- 结果标准化

## 实践建议

### 选择指南

**选择 claude-code-harness 如果**：
- 项目规模中小
- 需要快速上手
- 强调质量保证
- 团队规模较小

**选择 refact 如果**：
- 需要深度 IDE 集成
- 多语言项目
- 需要丰富工具支持
- 追求端到端自动化

**选择 agent-os 如果**：
- 企业级项目
- 团队协作开发
- 需要严格规范
- 长期维护产品

### 混合使用策略

可以结合多个项目的优点：

**基础架构**：采用 agent-os 的分层架构
**工作流**：借鉴 claude-code-harness 的 Plan→Work→Review
**工具集成**：参考 refact 的 LSP 集成方式

### 实现优先级

**第一阶段（MVP）**：
1. 实现基本的 Plan→Work→Review 循环
2. 简单的质量验证机制
3. 基础的上下文管理

**第二阶段（增强）**：
1. 添加工具集成
2. 改进评估机制
3. 增强错误处理

**第三阶段（优化）**：
1. 模块化工作流
2. 规范化管理
3. 性能优化

## 关键技术要点

### 1. 提示工程

**有效的提示结构**：
```
角色定义 + 任务描述 + 约束条件 + 输出格式 + 示例
```

**示例**：
```
你是一个资深的 Python 后端开发专家。

任务：实现用户认证 API

约束：
- 使用 FastAPI 框架
- 密码必须 bcrypt 加密
- 返回 JWT token
- 完整错误处理
- 包含单元测试
```
